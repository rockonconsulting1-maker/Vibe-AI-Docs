# ⚡ TRIGGERS

---

## 👤 Contact Triggers

| Trigger | Description | Filters Available | Notes |
|---|---|---|---|
| **Contact Created** | Fires when a new contact record is created | Contact fields, tags, source | Works with manual creation, imports, and form submissions |
| **Contact Tag Added** | Fires when a specific tag is added to a contact | Tag name | Can be triggered by manual, workflow, CSV import, or 3rd party |
| **Contact Tag Removed** | Fires when a specific tag is removed from a contact | Tag name | Tag removal via 3rd party integrations not always supported |
| **Contact Changed** | Fires when any contact field value is updated | Field name, field value | Useful for lifecycle stage automation |
| **Contact DND Updated** | Fires when a contact's Do Not Disturb status is changed | DND status, channel | Allows compliance-based automation |

---

## 📅 Appointment / Calendar Triggers

| Trigger | Description | Filters Available | Notes |
|---|---|---|---|
| **Customer Booked Appointment** | Fires when a contact books an appointment | Calendar name | Triggers at booking time, before the appointment occurs |
| **Appointment Status Changed** | Fires when an appointment status changes | Status (New, Confirmed, Showed, No-Show, Cancelled, Invalid), Calendar name | Most commonly used for no-show recovery and confirmation workflows |

## 🏆 Opportunity / Pipeline Triggers

| Trigger | Description | Filters Available | Notes |
|---|---|---|---|
| **Opportunity Created** | Fires when a new opportunity is added to a pipeline | Pipeline, stage, standard & custom opportunity fields | Great for new deal onboarding sequences |
| **Opportunity Status Changed** | Fires when the status of an opportunity is updated (open, won, lost, abandoned) | Status value, pipeline, standard & custom fields | Use for win/loss follow-up sequences |
| **Pipeline Stage Changed** | Fires when an opportunity moves to a new pipeline stage | Pipeline name, stage name, standard & custom fields | Primary trigger for deal progression automation |
| **Stale Opportunity** | Fires when an opportunity has had no activity for a set number of days | Number of days inactive, pipeline, stage, fields | Ideal for re-engagement or pipeline hygiene workflows |

## 💳 Payment / Order Triggers

| Trigger | Description | Filters / Sub-filters | Notes |
|---|---|---|---|
| **Payment Received** | Fires on any incoming payment | Payment Source (Invoice, Funnel/Website, Calendar), Sub-source (Text2Pay, One-time, Recurring, Upsell), Transaction Type (Customer present / not present), Calendar name, Global Product, Payment Status (Success / Failed) | Covers one-time, subscription, invoice, and text-to-pay payments |
| **Order Submitted** | Fires when an order is placed (after payment finalizes) | Order source, product | Suitable for post-purchase workflows |
| **Order Form Submission** | Fires when an order form is submitted (fires before payment processing) | Form name | Note: fires before payment is confirmed |
| **Invoice Trigger** | Fires on invoice lifecycle events | Invoice status (sent, paid, etc.) | Automate invoice follow-ups and receipts |
| **Subscription Trigger** | Fires on subscription status changes | Subscription status (Created, Trial-to-Active, Cancelled), product | Compatible with Stripe, Authorize.net, NMI; PayPal limited to creation only |
| **Refund Trigger** | Fires when a refund is processed | Refund success / failure, full / partial refund, refund amount, refund source | Branch-based logic available for different refund scenarios |

## 📝 Notes & Best Practices

- **Workflow types matter:** Contact-based, Company-based, and Custom Object-based workflows have different available triggers and actions. Always select the correct workflow type for your use case.
- **Custom Object and Company workflows** require those records to exist before automation can run.
- **Facebook/Instagram messaging** actions are governed by platform-level messaging windows (24-hour rules). Plan accordingly.
- **Inbound Webhook trigger** requires either an email address or phone number in the payload to create or find a contact.
- **Clear Field Data** only works on custom contact fields — standard fields (first name, email, phone) cannot be cleared this way.
- **Array Functions** are designed for workflows receiving array data from Payment triggers, Webhook triggers, or other structured data sources.
- **Manual SMS / Manual Call** actions pause the workflow until a human team member completes the action from the Conversations > Manual Actions queue.
- **Drip action** is used to control the rate of contacts passing through bulk workflow enrollment — prevent server overload and spam flags.
- **All trigger filters support AND/OR logic** for precision targeting within a single trigger.
- **Premium execution credits reset monthly** at payment time; unused credits do not roll over (except during mid-cycle plan upgrades).
